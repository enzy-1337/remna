## 1. Целевая логика (из мастер-плана, §2.9)

Смысл для продукта:

1. Пока **не открыт** текущий оплаченный блок 1 ГБ: при **факте использования трафика** (первый байт / событие от панели — *уточнить по типу вебхуков*) **сразу списать 5 ₽** за блок «до 1 ГБ».
2. Следующие **5 ₽** — при потреблении **сверх** уже оплаченного целого гигабайта (ступени 1→2→3 ГБ относительно **оплаченных** блоков).
3. При активном **пакете** сначала расходовать **пакетный лимит ГБ**, затем pay-as-you-go.

Это можно представить как состояния на стороне **биллинга** (абстрактно):

| Состояние (абстрактно) | Описание |
|-------------------------|----------|
| `P` | Есть пакетный остаток ГБ в текущем периоде — трафик съедает пакет **без** рублёвого шага (или по правилам пакета). |
| `F0` | Payg, **ещё не оплачен** текущий блок «до 1 ГБ» (ожидается первое событие потребления). |
| `Fk` | Payg, оплачено **k** целых гигабайт-блоков вперёд; трафик в пределах оплаченного — без нового списания; при пересечении границы **k→k+1** — списание **5 ₽**. |

*Точная привязка к событиям панели 2.7.4 — внешний контракт; таблица ниже фиксирует **текущую** реализацию в remna.*

## 2. Текущая реализация в коде (Remnawave → remna)

**Источник событий:** вебхук `traffic.gb_step` → `webhook_ingress_service.process_remnawave_event` → `rating_service.charge_gb_step`.

**Пакет (`Plan.is_package_monthly`, лимит `monthly_gb_limit`):**

- Считается число уже записанных событий `traffic_gb_step` за **календарный месяц в `BILLING_CALENDAR_TIMEZONE`**, содержащий `event_ts` (`billing_package_month_utc_bounds`).
- Если `used_steps_in_month < monthly_gb_limit` → шаг **покрыт пакетом**: пишется `BillingUsageEvent` с `meta.package_covered=true`, **рубли не списываются**.
- Иначе → списание `BILLING_GB_STEP_RUB` (+ опционально мобильная надбавка) с idempotency `gb:{event_id}`.

**Идемпотентность:** повтор того же `event_id` не создаёт второй `BillingUsageEvent` с тем же `event_id` (уникальность на уровне модели/БД — проверить миграции); ledger защищён ключом `gb:{event_id}`.

## 3. Таблица: событие → действие (как сейчас)

| Условие | Событие | Списание ₽ | Запись usage | Примечание |
|---------|---------|------------|--------------|------------|
| Не hybrid или v2 выкл | `traffic.gb_step` | 0 | нет | `ignored` на вебхуке. |
| Hybrid, пакет, шаг в пределах лимита | `traffic.gb_step` | 0 | да, `package_covered=true` | «Съедается» пакет. |
| Hybrid, шаг сверх пакета / без пакета | `traffic.gb_step` | `BILLING_GB_STEP_RUB` (+ mobile) | да | `apply_debit`; дневная сводка `gb_units++`. |
| Пол баланса | `traffic.gb_step` | 0 (reject) | нет | `apply_debit` не проходит → вебхук `rejected`, **usage не создаётся**. |

Для **payg**: сначала `apply_debit`, при успехе — `BillingUsageEvent` и дневная сводка. Для **пакета**: только usage с `package_covered=true`.

**Устройства (`device_daily`):** для платного пути применён тот же порядок — сначала `apply_debit`, затем запись `BillingUsageEvent` (пакетное покрытие — только usage, без списания).

## 4. Разрыв с §2.9 мастер-плана

| Аспект плана | Код сегодня |
|--------------|-------------|
| Явный учёт «оплаченных блоков вперёд» отдельно от шагов панели | Нет: **один шаг вебхука = один логический шаг** списания или покрытия пакетом. |
| «Первый байт» vs «шаг ГБ» | Не различается: только тип **`traffic.gb_step`**. Нужен ответ Q1 по OpenAPI/логам **2.7.4**. |
| Календарный месяц пакета | **Исправлено (фаза 1, часть):** счётчик шагов ГБ для пакета — в календарном месяце `BILLING_CALENDAR_TIMEZONE` (`billing_package_month_utc_bounds`), как детализация по суткам. |
