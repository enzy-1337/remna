## 1. Цель проекта (one-liner)

Переработать **remna** в систему управления VPN с **гибридной тарификацией**: месячные пакеты (лимит ГБ + устройств) + **pay-as-you-go** сверх лимита; баланс в рублях с полом; учёт трафика и устройств по **вебхукам Remnawave**; детализация в боте, web-admin и тикетах; рефералы и промокоды на одном балансе; режим «оптимизированный маршрут» через squad + доплата за ГБ; перевыпуск подписки одной операцией API панели.

## 2. Зафиксированные решения (против «забыли обсудить»)

### 2.1. База данных

- **Сейчас:** только **PostgreSQL** (как в репозитории). Миграции Alembic, `asyncpg`.
- **Позже:** MariaDB — строго по [mariadb-migration-future.md](./mariadb-migration-future.md), отдельным эпиком **после** стабилизации логики на Postgres.

### 2.2. П.3 — Telegram: polling vs webhook (**решение автора плана**)

| Аспект | Решение |
|--------|---------|
| **По умолчанию** | Оставить **long polling** (`aiogram` + `start_polling`), как сейчас: меньше инфраструктурных зависимостей (публичный URL только для Telegram, сертификаты, секрет пути, антидубликаты). |
| **Вебхуки Remnawave** | Остаются **основным** источником событий трафика/устройств (как договорено). |
| **Telegram webhook** | **Не внедрять в первой волне.** Вынести в **фазу «опционально»**: включение через `TELEGRAM_WEBHOOK_*` + эндпоинт в `api`, когда будет явная потребность (масштаб, хостинг без исходящих). |

Итого: **ничего не ломаем в проде по Telegram**; фокус — биллинг и панель.

### 2.3. П.12 — Калькулятор (**нужен по ТЗ**)

- В боте (и при желании дубль в web-admin): экран **«Калькулятор»**.
- Пользователь вводит (или выбирает слайсерами) **число ГБ за период** и **число устройств·дней** (или сценарий «N устройств × 30 дней»).
- Вывод:
  - оценка **pay-as-you-go** (трафик + устройства + при включённом режиме — доплата за ГБ «оптимизированного маршрута»);
  - блок **«Сравнение с тарифом»**: пользователь выбирает **конкретный план из админки** (активные `Plan`) → показать **цена пакета** vs **оценка pay-as-you-go** за тот же период/объём (как в примере из ТЗ: 1 устр + 15 ГБ → 75 + 75 = 150 ₽).
- Использовать существующую логику оценки где возможно: например `shared/services/billing_calculator.py` (`estimate_pay_per_use_30d_rub`) — расширить под сценарии и надбавку маршрута.

### 2.4. Источник правды по трафику

- **Вебхуки панели** — основной канал.
- **Опционально позже:** минутный reconcile (опрос API) только как **страховка** при пропуске вебхуков — не в фазе 0, описать в фазе N после метрик.

### 2.5. Баланс и рефералы

- **Один** счёт: `users.balance` (рубли). Реферальные 10% с пополнений рефералов — **транзакции того же типа зачисления**, без отдельного «бонусного кошелька».

### 2.6. Первое пополнение

- Акции на **любое первое** успешное пополнение; конкретные пороги (например «+100% при первом пополнении от 10 ₽») и **+5 ГБ** — параметризовать в `Settings` / админке, не зашивать магические числа в хендлерах.

### 2.7. Устройства в один день

- Повторное добавление «того же» устройства в календарный день — **без второго списания за день**.
- Идентичность: **HWID + стабильные ключи из панели + имя устройства** (уточнить в реализации маппинг полей из payload вебхуков); идемпотентный ключ списания хранить в `BillingUsageEvent.event_id` / аналог.

### 2.8. «Оптимизированный маршрут»

- В нашей БД — **флаг пользователя** (или настройка подписки — уточнить в фазе модели).
- В Remnawave — **переключение squad** (UUID из конфига).
- Тарификация: **+2,5 ₽ за каждый оплачиваемый ГБ** в этом режиме (в дополнение к базовым 5 ₽/ГБ или к пакету — формализовать в матрице списаний в фазе 1).

### 2.9. Модель ГБ (п.5 из переписки)

Смысл (для реализации в коде, формулы — в отдельной подзадаче):

- Пока **не открыт** текущий оплаченный блок 1 ГБ: при **факте использования трафика** (первый байт / событие от панели — уточнить по типу вебхуков) **сразу списать 5 ₽** за блок **«до 1 ГБ»**.
- Следующие **5 ₽** — при потреблении **сверх** уже оплаченного целого гигабайта (ступени 1→2→3 ГБ относительно оплаченных блоков).
- Не смешивать с «пакетными» ГБ без явного правила: при активном пакете сначала **съедать пакетный лимит**, затем pay-as-you-go (согласовано в п.11 переписки).

## 4. Нефункциональные рекомендации

1. **Идемпотентность** всех списаний и вебхуков — не дублировать при ретраях панели (уже есть `RemnawaveWebhookEvent`, `BillingUsageEvent.event_id` — сохранять подход).
2. **Не раздувать scope:** одна фаза = один мержируемый результат + тесты.
3. **Отрицательный баланс и отключение:** единая политика (порог, уведомления, снятие squads) — сверяться с `billing_v2` / `negative_balance_notify_loop` и не плодить второй «движок».
4. **Логи и метрики:** при новых ветках биллинга — структурированный лог `user_id`, `event_id`, сумма.
5. **Документация API панели:** [Remnawave API](https://docs.rw/api) — для перевыпуска подписки искать **один** эндпоинт и зафиксировать его имя в этом файле после проверки на вашей версии панели.

- Детализация дня/месяца в боте; в тикетах — день.
- Калькулятор с блоком «сравнение с выбранным тарифом».
- Один баланс; реферал 10% на пополнения рефералов зачисляется на него.
- Режим «оптимизированный маршрут» включается и влияет на squad и на +2,5 ₽/ГБ.
- Перевыпуск подписки через **один** вызов API (после фиксации эндпоинта).
